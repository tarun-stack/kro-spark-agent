# Remediation Agent

Receives failure diagnoses from the healing agent and opens a GitHub PR with
a proposed code/config fix, using a small first-party OpenAI tool-calling
agent (`run_openai_agent` in `server.py` ‚Äî not the Claude Agent SDK, which is
Anthropic-only). **Its authority ends at the PR** ‚Äî merging, image build, and
deploy stay with humans + CI.

Deployment for this component (Deployment, Service, NetworkPolicy, ConfigMap) is
generated by the `SparkHealingAgent` KRO resource when `spec.remediation.enabled`
is `true` ‚Äî see `../manifests/rgd.yaml` and the top-level `../README.md`. This
directory only holds the application image itself.

## Flow

```
healing agent ‚îÄ‚îÄPOST /remediate‚îÄ‚îÄ‚ñ∂ remediation agent
                                     1. look up repo for app (own config)
                                     2. shallow-clone repo
                                     3. OpenAI tool-calling loop: locate bug, edit, run tests
                                        (list_files/read_file/write_file/run_command, all
                                        path-confined to the checkout)
                                     4. branch + commit + push (deterministic code, not the model)
                                     5. gh pr create  ‚îÄ‚îÄ‚ñ∂  human review ‚îÄ‚îÄ‚ñ∂ CI build ‚îÄ‚îÄ‚ñ∂ deploy
```

## Build

```bash
docker build -t spark-remediation-agent:latest kro-spark-agent/remediation-agent/
kind load docker-image spark-remediation-agent:latest --name claude-demo
```

The GitHub token must be a **fine-grained PAT (or GitHub App token) with only
`contents: write` + `pull_requests: write` on only the Spark job repo(s)**.
No `workflow` scope, no org access. Set it, along with `OPENAI_API_KEY`, in
the `remediation-credentials` Secret (see `../manifests/secrets.example.yaml`),
and the app‚Üírepo mapping in `spec.remediation.repoMapJson` on the
`SparkHealingAgent` instance.

## Test without a real failure

```bash
kubectl port-forward svc/spark-healer-remediation 8081:8080 -n kro-spark-agent &
curl -s localhost:8081/remediate -X POST -H 'Content-Type: application/json' -d '{
  "app_name": "test-failing-job",
  "root_cause": "mainClass org.example.DoesNotExist not present in the jar",
  "summary": "wrong mainClass in job spec",
  "log_excerpt": "java.lang.ClassNotFoundException: org.example.DoesNotExist"
}'
```

## Security posture

- `automountServiceAccountToken: false` ‚Äî zero Kubernetes API access; it runs
  untrusted repo tests, so it must hold no cluster credentials.
- App‚Üírepo mapping from ConfigMap only; the caller can't choose the repo.
- Log excerpts are labeled untrusted in the prompt and quoted in the PR body.
- Git/PR operations are deterministic server code; the model only edits files.
- Every tool call's path is resolved with `os.path.realpath` and rejected if
  it escapes the checkout directory.
- Per-(app, root-cause) cooldown prevents PR spam from crash-looping jobs.
- NetworkPolicy: ingress from healing agent only; egress DNS + 443 only
  (hostname-level egress needs Cilium/Calico; inert on plain kind/Docker Desktop).
- **Branch protection on the target repo is the real enforcement** ‚Äî require
  review + passing CI, never grant the token auto-merge.
