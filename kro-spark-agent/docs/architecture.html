Who can do what

read-only

healing-agent

Watches SparkApplication status, reads driver logs, asks OpenAI to diagnose a failure. Cannot patch or delete anything — every action is a proposal sent elsewhere.

write, sole holder

actuator

The only identity in the system with patch/delete on SparkApplications. Re-derives executor counts and restart history itself rather than trusting the caller's numbers.

zero credentials

remediation-agent

Edits a cloned repo with an OpenAI tool-calling loop, confined to the checkout. automountServiceAccountToken: false — no Kubernetes API access at all.

read-only

proxy-sidecar

All egress from the healer Pod is iptables-redirected here. Only api.openai.com gets the real key injected; the Kubernetes API, the actuator, and SMTP pass through untouched.

Reading the boundaries

→Diagnose vs. act. The healing agent can see everything and change nothing; the actuator can change one thing (SparkApplications in spark-jobs) and see nothing else. Verified live: kubectl auth can-i delete sparkapplications --as=healer → no, --as=actuator → yes.
→A human is the approval gate, twice over. Restart/scale needs a clicked email link; a code fix needs a merged PR. Neither path can complete itself.
→The proxy's job is narrower than it looks. It only ever protects one secret — the OpenAI key. Everything else crossing that sidecar (Kubernetes API calls, actuator calls, SMTP) is passed through in the clear, because none of it presents a hostname the proxy can act on the way HTTPS does.
