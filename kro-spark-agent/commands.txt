1. CREATE THE KIND CLUSTER
--------------------------------------------------------------------------
kind create cluster --name claude-demo --wait 60s
kubectl config current-context   # should print: kind-claude-demo


--------------------------------------------------------------------------
2. INSTALL KRO (the ResourceGraphDefinition controller)
--------------------------------------------------------------------------
helm install kro oci://registry.k8s.io/kro/charts/kro \
  --namespace kro-system --create-namespace --insecure-skip-tls-verify
kubectl wait --for=condition=Available deploy -n kro-system --all --timeout=90s
kubectl get pods -n kro-system


--------------------------------------------------------------------------
3. INSTALL THE SPARK OPERATOR
--------------------------------------------------------------------------
kubectl create namespace spark-jobs

helm repo add spark-operator https://kubeflow.github.io/spark-operator
helm repo update spark-operator

# IMPORTANT: the value key is "spark.jobNamespaces" (nested), NOT
# "sparkJobNamespaces" or top-level "jobNamespaces" — get this wrong and the
# controller silently only watches the "default" namespace.
helm install spark-operator spark-operator/spark-operator \
  --namespace spark-jobs \
  --version 2.5.1 \
  --set "spark.jobNamespaces={spark-jobs}"

kubectl wait --for=condition=Available deploy -n spark-jobs --all --timeout=90s
kubectl get pods -n spark-jobs
kubectl get crd | grep -i spark

# Sanity check — must show "spark-jobs", not "default":
kubectl get pods -n spark-jobs -l app.kubernetes.io/name=spark-operator \
  -o jsonpath='{.items[0].spec.containers[0].args}' | tr ',' '\n' | grep namespaces

# The chart auto-creates the driver ServiceAccount "spark-operator-spark"
# with the RBAC Spark drivers need to launch executor pods. Verify it exists
# (do NOT create it manually — that causes a Helm ownership conflict):
kubectl get sa spark-operator-spark -n spark-jobs

4. BUILD THE IMAGES
--------------------------------------------------------------------------
docker build -t credential-proxy:latest --build-arg GOPROXY=direct proxy/
docker build -t spark-healing-agent:latest kro-spark-agent/healing-agent/
docker build -t spark-actuator:latest kro-spark-agent/actuator/
docker build -t spark-remediation-agent:latest kro-spark-agent/remediation-agent/   # optional, only if using remediation

================================================================================
TROUBLESHOOTING NOTES (things that actually broke during setup)
================================================================================
- Spark Operator only watching "default": you set --set "jobNamespaces=..."
  instead of --set "spark.jobNamespaces=...". Fix with `helm upgrade` using
  the correct nested key, then delete/reapply the test SparkApplications.

- "invalid ownership metadata" on `helm install spark-operator`: something
  (you, a previous run) manually created serviceaccount/spark-operator-spark
  outside Helm. Delete it (and any Role/RoleBinding you made for it) before
  installing — the chart creates its own with correct labels.

- Healing-agent pod stuck in Init:Error on iptables-init: CRLF line endings
  in proxy/*.sh from a Windows checkout broke `set -euo pipefail`. Already
  fixed in proxy/Dockerfile (strips \r at build time) — rebuild the image if
  you still see this.

- Any in-cluster HTTP call through the proxy sidecar (to the actuator or
  remediation service) times out / hangs forever: this was a real bug in
  proxy/main.go's extractHTTPHost() — br.Peek(4096) blocked indefinitely for
  any HTTP request shorter than 4096 bytes (i.e. almost all of them), since
  no read deadline was set. Already fixed (bounded with a 1s deadline,
  same pattern as the TLS SNI peek) — rebuild credential-proxy:latest and
  kind load it if you still see hangs.

- SMTP login fails with "SmtpClientAuthentication is disabled for the
  Tenant": your email provider (commonly M365) has basic SMTP AUTH turned
  off tenant-wide. Not fixable from this code — either get an admin
  exception ("Authenticated SMTP" in the Exchange admin center) or point
  spec.smtpHost/smtpHost credentials at a personal Gmail account instead.

- "Credit balance is too low" from the `claude` CLI or Anthropic API: this
  system's OpenAI-based components (diagnosis, remediation) are unaffected
  — this only matters if you reintroduce anything Anthropic-backed.
